# Phase 5: PR (Sprint Lead solo)

Create a PR from the sprint branch to the base branch.

## Workflow

1. **Pre-check**: review changes on the sprint branch in each role worktree.
1.5. **E2E Full-Suite Regression Gate** (when the app worktree has changes):
   - Confirm the user has the simulator/emulator ready.
   - Run: `cd app && yarn workspace <example app> e2e:auth` (full suite, e.g., 22+ flows).
   - Result:
     - **All PASS** → Proceed with PR creation. Record "Full suite PASS ({N} flows)" in the PR body's `### E2E` section.
     - **Some FAIL** → Block PR creation. Print failed flow list, then present user options:
       a) Return to Phase 4 (additional fix loop) — recommended.
       b) Force progress with `--allow-e2e-fail` (list failed flows in PR body).
       c) If environment issue (token/simulator), recover and rerun.
   - **Skip conditions**: `--skip-e2e` flag specified, or app has no changes.
2. **Collect task statuses**: list of COMPLETED/FAILED tasks.
3. **Push + create PR** (per role, **user confirmation required**):

   `{branch_prefix}` is set in sprint-config.yaml; the default is `sprint`.

```bash
cd {role-dir}           # e.g., backend, app
git checkout {branch_prefix}/{sprint-id}
git push -u origin {branch_prefix}/{sprint-id}
```

**PR body/title safety rule**: For any dynamic field substituted into the body (`{PRD scope summary}`, `{objective}`, `{eval result}`, etc.), pre-replace **`"` → `'`** and **`` ` `` → `'`**. Apply the same to the title. The HEREDOC itself uses `<<'EOF'` (quoted heredoc), so bash expansion is safe; however, if downstream CI templates reuse the squash commit message via `git commit -m "${{ inputs.commit_message }}"`, embedded `"` characters may break bash quoting and fail deploys. Apply this rule as a mitigation.

```bash
gh pr create \
  --base {base-branch} \
  --head {branch_prefix}/{sprint-id} \
  --title "feat: Sprint {sprint-id} — {role}" \
  --body "$(cat <<'EOF'
## Sprint: {sprint-id}

### Summary
{PRD scope summary}

### Tasks
| Task | Status | Evaluation |
|------|--------|------------|
| {task-id} | {status} | {eval result} |

### Evaluation Summary
- Groups: {N} total, {N} accepted, {N} failed
- Issues found & fixed: {N}

### Generated by
zachflow Harness-Driven Sprint System
EOF
)"
```

4. **Duplicate PR check**: if a PR with the same head/base already exists, surface the existing PR URL.

## Output

When the Gate passes:
1. Create the checkpoint file (`checkpoints/phase-5-summary.md`).
2. **Print Sprint Status** — emit the `--status` dashboard to show current progress.
3. Enter the next Phase.

```
Sprint PR: {sprint-id}
  backend:  {url} ({branch_prefix}/{sprint-id} → {base})
  app:      {url} ({branch_prefix}/{sprint-id} → {base})

Sprint pipeline complete!

[Sprint Status Dashboard]

→ Proceeding to Phase 6: Retrospective
```
