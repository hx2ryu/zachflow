# Evaluation Criteria — Sprint {sprint-id}

> Generated by the Planner in Phase 2 (Spec). Frames the Evaluator's grading rubric.

## Grading Dimensions

| Criterion | Weight | Description |
|-----------|--------|-------------|
| **Correctness** | High | Does the business logic match the ACs exactly? |
| **Completeness** | High | Are all requirements implemented (no stubs/placeholders)? |
| **Edge Cases** | Medium | Are boundary conditions and error states handled correctly? |
| **Integration** | Medium | Is the actual implementation consistent with the API contract? |
| **Code Quality** | Low | Adherence to existing patterns, readability (the model handles this baseline well) |

> Code Quality is weighted low because Claude already produces high-quality code by default.
> Spending energy on Correctness and Completeness yields more quality improvement.

## Severity Classification

| Severity | Definition | Examples |
|----------|------------|----------|
| **Critical** | Function broken or data integrity at risk | API returns 500, infinite loop, injection vulnerability |
| **Major** | Function works but violates ACs / business rules | Unfollow doesn't decrement count, blocked user appears in feed |
| **Minor** | Code-quality issue with no behavior impact | Unused import, inefficient query |

## Verdict Rules

- **PASS**: Critical 0, Major 0 → proceed to next group
- **ISSUES**: Critical 0, Major 1+ → fix and re-evaluate
- **FAIL**: Critical 1+, or Major 3+ → fix and re-evaluate

## Evaluator Calibration

### Skepticism Anchors

```
You are not a code reviewer — you are a bug hunter.

- Doubt implementations even when they look complete.
- If you feel "this should work...", trace it for real.
- Once you find an issue, do not talk yourself out of it.
- Start with edge cases, not the happy path.
- "Code exists" and "code works correctly" are different things.
```

### Anti-Patterns to Avoid

- Marking VERIFIED based only on file/function existence
- Listing issues then concluding "overall well implemented"
- Charitably interpreting Generator intent to pass incomplete work
- Testing only the happy path before declaring PASS
